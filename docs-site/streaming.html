Live Streaming

Fabriq provides a complete live streaming infrastructure as a built-in package. It replaces expensive hosted signaling services (Agora, LiveKit) and media servers by running the control plane in PHP and delegating the media plane to FFmpeg — all within the same Swoole process.

Install via Composer: composer require fabriq/streaming — this pulls in fabriq/kernel, fabriq/storage, and fabriq/observability automatically. View on Packagist →
Opt-in: Streaming is disabled by default. To enable it: 1) set STREAMING_ENABLED=1 environment variable (or 'enabled' => true in config/streaming.php), and 2) uncomment StreamingServiceProvider in config/app.php.
Cost Savings: By combining signaling, chat, and stream management into a single Fabriq process, you eliminate $49–$500/month in third-party signaling costs and $20–$100/month for a separate chat server.

Architecture Overview

The streaming package uses a hybrid approach:

ComponentResponsibilityRuns In
SignalingHandlerWebRTC SDP/ICE exchangeSwoole WS Worker
StreamManagerStream lifecycle (create, start, end)Swoole Worker
TranscodingPipelineFFmpeg process managementChild Process
HlsManagerHLS manifest + segment servingSwoole HTTP Worker
ViewerTrackerConcurrent viewer countingRedis
ChatModeratorChat slow mode, filters, bansRedis + Worker
StreamRepositoryPersistent stream dataMySQL via pool

For small audiences (≤50 viewers), WebRTC P2P is used — the media flows directly between the streamer and viewers, with Fabriq only handling the signaling. For large audiences, the TranscodingPipeline kicks in, using FFmpeg to produce HLS segments that can be served via CDN to thousands of viewers.

Configuration

All streaming settings live in config/streaming.php:

<?php
// config/streaming.php
return [
    'enabled' => true,
    'ffmpeg_path' => '/usr/bin/ffmpeg',

    'hls' => [
        'segment_duration' => 4,
        'playlist_size' => 5,
        'storage_path' => '/tmp/fabriq-hls',
    ],

    'stream_key_ttl' => 86400,
    'max_concurrent_transcodes' => 4,

    'chat' => [
        'slow_mode_seconds' => 0,
        'max_message_length' => 500,
    ],
];
Tip: Enable the StreamingServiceProvider in config/app.php to activate streaming features. It’s included by default but gated by the streaming.enabled config flag.

Stream Lifecycle

Creating a Stream

Streams are created via the REST API or directly through the StreamManager:

// Via StreamManager
$streamManager = $container->make(StreamManager::class);
$result = $streamManager->createStream(
    tenantId: 'acme',
    userId: 'user_123',
    title: 'My Live Stream',
);
// Returns: ['stream_id' => 'stream_abc123', 'stream_key' => 'sk_...']

Or via the REST API:

# Create a stream
curl -X POST http://localhost:8000/api/streams \
  -H "Content-Type: application/json" \
  -d '{"title": "My Live Stream"}'

# Response:
# {"stream_id": "stream_abc123", "stream_key": "sk_..."}

Stream States

StateDescription
pendingStream created, stream key issued, waiting for streamer to connect
liveStreamer is broadcasting, viewers can join
endedStream has ended, cleanup in progress

WebRTC Signaling

The SignalingHandler replaces hosted signaling services by handling SDP offer/answer and ICE candidate exchange over WebSocket:

Signaling Flow

  1. Streamer sends offer with SDP → stored by Fabriq, relayed to viewers
  2. Viewer sends answer with SDP → relayed back to streamer
  3. ICE candidates exchanged similarly in both directions
  4. P2P media connection established — no Fabriq bandwidth used

Signaling Messages

// Streamer sends SDP offer
{"type": "offer", "stream_id": "stream_abc", "stream_key": "sk_...", "sdp": "v=0\r\n..."}

// Viewer subscribes to a stream
{"type": "subscribe", "stream_id": "stream_abc"}

// Viewer sends SDP answer
{"type": "answer", "stream_id": "stream_abc", "sdp": "v=0\r\n..."}

// ICE candidate exchange
{"type": "candidate", "stream_id": "stream_abc", "candidate": "candidate:..."}

Handling Signaling in Your WS Handler

use Fabriq\Streaming\SignalingHandler;

// In your WebSocket message handler:
$signaling = $container->make(SignalingHandler::class);
$data = json_decode($frame->data, true);

if (($data['type'] ?? '') === 'offer'
    || ($data['type'] ?? '') === 'answer'
    || ($data['type'] ?? '') === 'candidate'
    || ($data['type'] ?? '') === 'subscribe'
) {
    $signaling->handle($server, $frame, $tenantId, $userId);
}

HLS Transcoding

For large audiences, the TranscodingPipeline spawns FFmpeg processes to transcode incoming streams to HLS format:

use Fabriq\Streaming\TranscodingPipeline;

$transcoder = $container->make(TranscodingPipeline::class);

// Start transcoding (input from RTMP/WHIP ingest)
$transcoder->start('stream_abc', 'rtmp://localhost/live/stream_abc');

// Check if active
$transcoder->isActive('stream_abc'); // true

// Stop transcoding
$transcoder->stop('stream_abc');

// Clean up HLS files
$transcoder->cleanup('stream_abc');

HLS Playback

HLS segments are automatically served via HTTP:

# Get the HLS manifest
curl http://localhost:8000/hls/stream_abc/playlist.m3u8

# Video.js or HLS.js can play this URL directly:
# http://localhost:8000/hls/stream_abc/playlist.m3u8
CDN-Ready: HLS segments are served with appropriate Cache-Control headers. Manifests are no-cache (dynamic), while .ts segments are immutable (cacheable forever). Put a CDN in front for massive scalability.

Viewer Tracking

The ViewerTracker uses Redis sorted sets with TTL-based expiry for accurate concurrent viewer counts:

use Fabriq\Streaming\ViewerTracker;

$tracker = $container->make(ViewerTracker::class);

// Heartbeat (call every 15 seconds)
$tracker->heartbeat('acme', 'stream_abc', 'user_123');

// Get viewer count
$count = $tracker->count('acme', 'stream_abc');

// Get all viewer IDs
$viewers = $tracker->getViewers('acme', 'stream_abc');

// Remove viewer on disconnect
$tracker->remove('acme', 'stream_abc', 'user_123');

Chat & Moderation

The ChatModerator provides built-in moderation features for live stream chat:

use Fabriq\Streaming\ChatModerator;

$chat = $container->make(ChatModerator::class);

// Validate a message before sending
$result = $chat->validate('acme', 'stream_abc', 'user_123', 'Hello!');
// ['allowed' => true, 'reason' => '']

// Ban a user (0 = permanent, otherwise seconds)
$chat->ban('acme', 'stream_abc', 'troll_user', 3600);

// Unban
$chat->unban('acme', 'stream_abc', 'troll_user');

// Add word filter
$chat->addFilteredWords('acme', 'stream_abc', ['spam', 'scam']);

// Set slow mode (seconds between messages)
$chat->setSlowMode(5);
FeatureDescription
Slow ModeMinimum delay between messages per user (configurable)
Max LengthMaximum characters per message (default: 500)
Word FilterBlock messages containing banned words (per-stream)
Ban ListBlock specific users from chatting (with optional duration)

Service Provider

The StreamingServiceProvider registers all streaming services automatically when config('streaming.enabled') is true:

// config/app.php
'providers' => [
    // ... other providers
    \App\Providers\StreamingServiceProvider::class,
];

It registers:

  • StreamManager — Stream lifecycle management
  • SignalingHandler — WebRTC signaling over WebSocket
  • TranscodingPipeline — FFmpeg process management
  • HlsManager — HLS manifest & segment serving
  • ViewerTracker — Redis-backed viewer counting
  • ChatModerator — Chat moderation features

It also automatically registers the /hls/{streamId}/{filename} HTTP route for HLS playback.
