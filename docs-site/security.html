Security

Fabriq includes a complete security stack: JWT and API key authentication, RBAC + ABAC authorization via a policy engine, and Redis-backed rate limiting.

JWT Authentication

Fabriq includes a pure-PHP JWT implementation supporting HS256, HS384, and HS512 algorithms.

Issuing Tokens

use Fabriq\Security\JwtAuthenticator;

$jwt = new JwtAuthenticator(secret: 'my-secret', defaultTtl: 3600);

$token = $jwt->encode([
    'sub'       => $userId,
    'tenant_id' => $tenantId,
    'roles'     => ['admin', 'member'],
]);
// Returns: "eyJhbGci..."

Decoding Tokens

Decoding is typically handled automatically by AuthMiddleware, but you can also decode manually:

$claims = $jwt->decode($token);
// ['sub' => '...', 'tenant_id' => '...', 'roles' => [...], 'iat' => ..., 'exp' => ...]
// Returns null if invalid or expired.

After AuthMiddleware runs, the actor is available on the Context:

$actorId = Context::actorId();           // 'user-uuid'
$roles   = Context::getExtra('roles');   // ['admin']

JWT Claims

ClaimDescription
subSubject — the user/actor ID
tenant_idTenant ID (used by token tenant resolver)
rolesArray of role names for authorization
iatIssued at timestamp (auto-set)
expExpiration timestamp (auto-set from TTL)

API Key Authentication

API keys use the format sf_{prefix}_{secret}. The prefix is stored in plaintext for fast lookup, while the full key is hashed with SHA-256.

Generating Keys

use Fabriq\Security\ApiKeyAuthenticator;

// Generate a new key
$key = ApiKeyAuthenticator::generateKey();
// ['key' => 'sf_a1b2c3d4_...', 'prefix' => 'a1b2c3d4', 'hash' => 'sha256...']
// Store prefix + hash in the api_keys table; give the full key to the user.

Creating the Authenticator

// With DB-backed lookup
$apiKeyAuth = ApiKeyAuthenticator::withDb($dbManager);

// Or with a custom lookup function
$apiKeyAuth = new ApiKeyAuthenticator(function (string $prefix) use ($repo): ?array {
    return $repo->findApiKeyByPrefix($prefix);
});
Tip

API keys are validated via the X-API-Key header or the Authorization: Bearer header. The AuthMiddleware detects the format automatically.

RBAC + ABAC Policy Engine

The PolicyEngine combines Role-Based Access Control (RBAC) with Attribute-Based Access Control (ABAC) for fine-grained authorization.

Defining Roles

use Fabriq\Security\PolicyEngine;

$engine = new PolicyEngine();

// Define roles with permissions (format: "resource:action")
$engine->defineRole('admin', ['*:*']);                          // full access
$engine->defineRole('editor', ['posts:create', 'posts:read', 'posts:update']);
$engine->defineRole('viewer', ['posts:read']);

ABAC Conditions

Add attribute-based conditions that are checked after RBAC allows:

$engine->addCondition('posts', 'delete', function (array $ctx): bool {
    // Only allow deletion during business hours
    $hour = (int) date('G');
    return $hour >= 9 && $hour < 17;
});

Evaluating Access

$allowed = $engine->evaluate(
    actorId: 'user-123',
    roles: ['editor'],
    resource: 'posts',
    action: 'create',
    context: ['tenant_id' => '...', 'ip' => '...'],
);
// true

Decision Flow

RBAC check → if denied → DENY
           → if allowed → ABAC conditions
                        → if any fails → DENY
                        → all pass → ALLOW

Built-in Roles

RolePermissions
admin*:*
editorrooms:create, rooms:update, messages:*
viewermessages:read, rooms:read

Wildcards: rooms:* (all room actions), *:* (superuser).

Audit Logging

Every policy decision is logged as structured JSON:

{
    "actor_id": "user-123",
    "resource": "rooms",
    "action": "create",
    "allowed": true,
    "rbac": true,
    "abac": true,
    "timestamp": 1708127400.123
}

Protecting Routes

use Fabriq\Http\Middleware\PolicyMiddleware;

$policy = new PolicyMiddleware($engine);
$policy->protect('POST', '/api/posts', 'posts', 'create');
$policy->protect('DELETE', '/api/posts/{id}', 'posts', 'delete');

$middlewareChain->add($policy);

Rate Limiting

Fabriq provides Redis-backed sliding-window rate limiting per tenant:

use Fabriq\Http\Middleware\RateLimitMiddleware;
use Fabriq\Security\RateLimiter;

$limiter = new RateLimiter($dbManager);

$rateLimitMiddleware = new RateLimitMiddleware(
    limiter: $limiter,
    maxRequests: 100,       // 100 requests
    windowSeconds: 60,      // per 60-second window
);

$middlewareChain->add($rateLimitMiddleware);
// Adds X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After headers automatically.

Response Headers

HeaderDescription
X-RateLimit-LimitMaximum requests allowed in the window
X-RateLimit-RemainingRequests remaining in the current window
X-RateLimit-ResetUnix timestamp when the window resets
Retry-AfterSeconds to wait (only on 429 response)

Middleware Chain Order

The recommended middleware ordering ensures each layer has the context it needs:

1. CorrelationMiddleware  → sets request/correlation IDs
2. AuthMiddleware         → validates JWT/API key
3. TenancyMiddleware      → resolves tenant
4. RateLimitMiddleware    → enforces rate limits
5. PolicyMiddleware       → RBAC/ABAC check
6. [route handler]
