Game Server

Fabriq includes a real-time game server engine that runs inside the same Swoole process as your HTTP and WebSocket handlers. It supports casual, .io-style, MMORPG, MOBA, and competitive games — all without a separate game server.

Install via Composer: composer require fabriq/gaming — this pulls in fabriq/kernel, fabriq/storage, fabriq/observability, and rybakit/msgpack automatically. View on Packagist →
Opt-in: Gaming is disabled by default. To enable it: 1) set GAMING_ENABLED=1 environment variable (or 'enabled' => true in config/gaming.php), 2) uncomment GamingServiceProvider in config/app.php, and 3) install composer require rybakit/msgpack.
Cost Savings: A typical Node.js game server costs $20–$100/month per instance. By running the game loop inside Fabriq, you eliminate this cost entirely while gaining coroutine-safe database access, multi-tenancy, and built-in matchmaking.

Game Loop & Tick Rates

The GameLoop runs as a Swoole\Timer (not sleep()) for precise timing. Different game types use different tick rates:

TypeTick RateIntervalUse Case
casual10 Hz100msCard games, trivia, turn-based
mmo20 Hz50msMMORPG, open-world, sandbox
realtime30 Hz~33ms.io games, action RPGs
moba45 Hz~22msMOBA, RTS, team arena
competitive60 Hz~16msFPS, fighting games
twitch128 Hz~7.8msCompetitive FPS, rhythm games, ultra-low-latency
 
// The game loop runs per-worker, ticking all active rooms:
// GameLoop.php (simplified)
Swoole\Timer::tick($intervalMs, function () {
    foreach ($this->activeRooms as $room) {
        $room->tick($deltaTime);           // Run game logic
        $deltas = $room->getDeltas();       // Get changed state only
        $this->broadcastState($room, $deltas); // Push to players
    }
});

Binary Protocol

The UdpProtocol handles message serialization for both WebSocket binary frames and UDP packets:

use Fabriq\Gaming\UdpProtocol;

$protocol = new UdpProtocol('msgpack'); // or 'json'

// Encode a state update
$payload = $protocol->stateUpdate('room_abc', $gameState, $seq);

// Encode player input
$payload = $protocol->playerInput('room_abc', ['dx' => 1, 'dy' => 0], $seq);

// Decode any message
$message = $protocol->decode($rawData);
// ['type' => 'player_input', 'room_id' => '...', 'data' => [...], 'seq' => 42, 'ts' => 1234567890.123]

use Fabriq\Gaming\GameRoomManager;
use App\Gaming\Handlers\BattleGameHandler;

$roomManager = $container->make(GameRoomManager::class);

$room = $roomManager->createRoom(
    tenantId: 'acme',
    tickType: 'realtime',         // 30 Hz
    maxPlayers: 16,
    tickHandler: BattleGameHandler::create(),

echo $room->getId(); // "room_abc123"

Custom Game Logic

Your game logic lives in a tick handler — a callable that receives the room and delta time each tick:

// app/Gaming/Handlers/BattleGameHandler.php
namespace App\Gaming\Handlers;

use Fabriq\Gaming\GameRoom;

final class BattleGameHandler
{
    public static function create(): callable
    {
        return function (GameRoom $room, float $deltaTime): void {
            $state = $room->getState();

            // Process player inputs
            foreach ($room->getPlayerIds() as $playerId) {
                $input = $state["input:{$playerId}"] ?? null;
                if ($input === null) continue;

                // Apply movement with server authority
                $pos = $state['positions'][$playerId] ?? ['x' => 0, 'y' => 0];
                $pos['x'] += ($input['dx'] ?? 0) * 200 * $deltaTime;
                $pos['y'] += ($input['dy'] ?? 0) * 200 * $deltaTime;

                $room->setState("positions.{$playerId}", $pos);
                $room->setState("input:{$playerId}", null);
            }

            // Check win conditions, collisions, etc.
        };
    }
}

Matchmaking

The Matchmaker uses Redis sorted sets for efficient skill-based matching:

use Fabriq\Gaming\Matchmaker;

$matchmaker = $container->make(Matchmaker::class);
$matchmaker->setPlayersPerMatch(4);

// Set callback for when a match is found
$matchmaker->onMatch(function (array $players) use ($roomManager) {
    $room = $roomManager->createRoom(
        tenantId: $players[0]['tenant_id'],
        tickType: 'realtime',
    );

    foreach ($players as $player) {
        $room->addPlayer($player['player_id'], $player['fd'], $player['player_id']);
        // Notify player via WebSocket
    }
});

// Start polling
$matchmaker->start('acme', pollIntervalMs: 1000);

// Queue a player
$matchmaker->queue('player_123', skillRating: 1500.0, tenantId: 'acme', fd: $fd);

The matchmaking algorithm:

  1. Players are added to a Redis ZSET with their skill rating as the score
  2. A background poll scans the ZSET every second
  3. For each player, nearby players (± rating_range) are found
  4. After expand_after_seconds, the range widens automatically
  5. After max_wait_seconds, the player is removed from the queue

Pre-Game Lobbies

The LobbyManager handles the waiting-room phase before a game starts:

use Fabriq\Gaming\LobbyManager;

$lobbyManager = $container->make(LobbyManager::class);

// Create lobby for a room
$lobbyManager->createLobby($room);

// Toggle player ready state
$lobbyManager->setPlayerReady($roomId, $playerId, ready: true);

// Lobby auto-starts when:
//   1. All players are ready
//   2. There are at least 2 players
//   3. A countdown timer (default 5s) expires

Lobby flow: Join → Ready Check → Countdown → Game Start

Player Reconnection

Players who disconnect can reconnect within a configurable grace period and resume their game:

use Fabriq\Gaming\PlayerSession;

$sessions = $container->make(PlayerSession::class);

// On connect
$sessions->connect('player_123', 'acme', 'user_123', $fd);

// On disconnect
$sessions->disconnect('player_123');

// On reconnect attempt
$result = $sessions->reconnect('player_123', $newFd);
if ($result['success']) {
    $roomId = $result['room_id']; // Resume in this room
}

// Check if reconnection is still possible
$sessions->canReconnect('player_123'); // true/false
Configurable window: Set reconnection_window_seconds in config/gaming.php. During this window, the player’s session, room, and state are preserved. Default: 30 seconds.

State Synchronization

The StateSync utility provides efficient delta compression — only changed state is sent to each player:

use Fabriq\Gaming\StateSync;

$sync = new StateSync();

// Register players
$sync->addPlayer('player_1');
$sync->addPlayer('player_2');

// Compute delta for a specific player
$delta = $sync->computeDelta('player_1', $currentState);
// Only contains keys that changed since player_1's last ack

// Player acknowledges receiving state up to seq #42
$sync->acknowledge('player_1', 42, $currentState);

// New/reconnecting player gets full state
$fullState = $sync->fullSync($currentState);

Service Provider

The GamingServiceProvider registers all gaming services when config('gaming.enabled') is true:

// config/app.php
'providers' => [
    // ... other providers
    \App\Providers\GamingServiceProvider::class,
];

It automatically:

  • Registers UdpProtocol, GameLoop, GameRoomManager, Matchmaker, PlayerSession, LobbyManager, StateSync
  • Starts the GameLoop per worker in onWorkerStart
  • Wires the UDP packet handler if server.udp_enabled is true
  • Registers GET /api/game/rooms for room discovery
