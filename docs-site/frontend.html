Frontend Serving

Fabriq serves per-tenant frontend builds directly through Swoole — no Nginx, no CDN required for development. Works with any framework (React, Vue, Svelte, Angular, etc.) and includes built-in CI/CD to clone, build, and deploy tenant frontends from Git.

Overview

Frontend serving turns Fabriq from a pure API backend into a unified application platform. The same Swoole process that handles your REST API and WebSocket connections also serves the HTML, CSS, and JavaScript that powers each tenant’s UI.

Browser → HTTP GET /dashboard
       → Fabriq Swoole Server
           → API prefixes (/api, /health, /metrics)? → Route to API handlers
           → Resolve tenant (domain_map → TenantResolver → DB lookup)
           → Try public/{tenant-slug}/dashboard → found? → sendfile()
           → SPA fallback → serve public/{tenant-slug}/index.html

Key capabilities:

  • Per-tenant builds — each tenant gets their own frontend in public/{tenant-slug}/
  • Custom domain resolution — 3-tier strategy: static config, TenantResolver, DB lookup
  • SPA fallback — serves index.html for client-side routing
  • Smart caching — fingerprinted assets get immutable/1-year headers, HTML gets no-cache
  • Build automation — built-in CI/CD clones Git repos, runs builds, deploys atomically
  • Zero config for APIs — paths starting with /api, /health, /metrics are never intercepted

Why Serve Frontends Through Fabriq?

Most platforms serve frontends through a separate web server (Nginx, Apache) or a dedicated hosting service (Vercel, Netlify, Cloudflare Pages). Fabriq takes a different approach: the same Swoole process that powers your API also serves your frontend. Here’s why that matters.

Simpler Infrastructure

Traditional SetupWith Fabriq
Nginx + PHP-FPM + CDN + CI/CD pipelineSingle Swoole process
Separate Nginx config per tenant (try_files, location blocks)Zero server config — tenants resolved automatically
Manage DNS, TLS certs, and CDN distribution for each custom domainCustom domains resolved via config, code, or DB — one TLS termination point
Deploy frontend and API independently, coordinate versionsFrontend and API ship together, atomic per-tenant deploys

No CORS, No Proxy Hops

When your frontend and API are served from the same origin, you eliminate an entire class of problems:

  • No CORS headers — frontend and API share the same origin, so browsers don’t block requests
  • No preflight requests — saves a round trip on every API call from the browser
  • No reverse proxy hop — requests go directly to Swoole instead of Nginx → PHP-FPM
  • Simpler cookies/sessions — same-origin means no SameSite or third-party cookie issues

Tenant-Aware by Default

In a traditional setup, the web server has no concept of tenants — it just serves files from a directory. With Fabriq:

  • The same tenant resolution logic (subdomain, header, JWT, custom domain) is shared between API and frontend serving
  • Adding a new tenant with a custom domain requires zero server reconfiguration — just add a row to the database
  • Suspended tenants can still see a custom suspension page from their frontend (Tier 3 DB lookup bypasses the strict TenantResolver)
  • Each tenant can use a completely different frontend framework — tenant A on React, tenant B on Vue, tenant C on plain HTML

High Performance

  • Swoole sendfile() — zero-copy file transfer at the kernel level, no PHP memory allocation
  • Fingerprint-based immutable caching — hashed filenames get 1-year Cache-Control headers automatically
  • No cold starts — Swoole is a long-running process, always ready to serve
  • Single port — HTTP API, WebSocket, and frontend all on port 8000 — simpler load balancing

Built-In CI/CD

Instead of configuring a separate CI/CD pipeline (GitHub Actions, Jenkins, etc.) for each tenant’s frontend:

  • Tenants store their Git repo URL in their config — Fabriq handles the rest
  • Builds are triggered via CLI, API, or webhook — no external pipeline to maintain
  • Atomic deployments ensure users never see a half-built state
  • Builds run in Swoole coroutines — non-blocking, no impact on API performance
When to use an external CDN instead

Fabriq’s built-in serving is ideal for SaaS platforms, internal tools, and moderate-traffic applications. For global consumer-facing apps with millions of users, you may want to put a CDN (CloudFront, Cloudflare) in front of Fabriq or serve static assets from object storage (S3). The two approaches aren’t mutually exclusive — Fabriq’s Cache-Control headers are CDN-ready out of the box.

Static File Serving

Enable frontend serving in config/static.php:

return [
    'enabled'            => true,
    'document_root'      => 'public',
    'multi_tenancy'      => true,
    'default_tenant_dir' => '_default',
    'spa_fallback'       => true,
    'index'              => 'index.html',
    'api_prefixes'       => ['/api', '/health', '/metrics'],
    'cache_max_age'      => 86400,
    'cors'               => false,
];

The StaticFileMiddleware is registered as the last route handler in the Application, meaning API routes always take priority. Only GET and HEAD requests are served as static files.

Directory Layout

public/
├── _default/                  # Fallback (login, marketing, 404)
│   ├── index.html
│   ├── assets/
│   │   ├── app.a1b2c3d4.js
│   │   └── style.e5f6g7h8.css
│   └── favicon.ico
├── acme/                      # Tenant "acme" build
│   ├── index.html
│   └── assets/
│       ├── app.x9y0z1w2.js
│       └── style.m3n4o5p6.css
└── globex/                    # Tenant "globex" build
    ├── index.html
    └── assets/

File Resolution Order

  1. Resolve the tenant slug from the request
  2. Try public/{tenant-slug}/{uri}
  3. Fall back to public/_default/{uri}
  4. If SPA fallback is on and no file matched, serve index.html from the resolved directory

MIME Types

The middleware includes a comprehensive MIME type map covering HTML, CSS, JS, JSON, SVG, images (PNG, JPEG, GIF, WebP, AVIF), fonts (WOFF, WOFF2, TTF, EOT), media (MP4, WebM, MP3, WAV), PDF, WebAssembly, and more.

Path Safety

Path traversal attacks are blocked — the middleware rejects any path containing .. or null bytes, and verifies that the resolved file is within the document root using realpath().

Per-Tenant Frontends

When multi_tenancy is enabled, the middleware resolves the tenant from the incoming request and serves files from their dedicated directory. If the tenant’s directory doesn’t exist, it falls back to _default/.

Request: GET /dashboard  (Host: acme.myapp.com)
  → Tenant resolved: "acme"
  → Try: public/acme/dashboard     → not found
  → SPA fallback: public/acme/index.html  → found → serve

Request: GET /login  (Host: unknown.com)
  → Tenant resolved: null
  → Try: public/_default/login     → not found
  → SPA fallback: public/_default/index.html  → found → serve
Framework Agnostic

Fabriq doesn’t care which frontend framework you use. Build your app with React, Vue, Svelte, Angular, SolidJS, or plain HTML — just drop the build output into the correct public/{tenant-slug}/ directory.

Custom Domains

Tenants can register custom domains (e.g., app.acme.com) that map to their frontend build. The StaticFileMiddleware resolves tenants via a 3-tier strategy:

Incoming request: Host: app.acme.com
  │
  ├─ Tier 1: domain_map config
  │    config/static.php → 'domain_map' → ['app.acme.com' => 'acme']
  │    Found? → slug = "acme"  ✓ (O(1), no DB)
  │
  ├─ Tier 2: TenantResolver
  │    Subdomain extraction, X-Tenant header, JWT claim
  │    Resolved? → slug = tenant->slug  ✓
  │
  └─ Tier 3: Database lookup
       ChatRepository::findTenantByDomain('app.acme.com')
       → SELECT * FROM tenants WHERE domain = 'app.acme.com'
       Found? → slug = row['slug']  ✓

  None matched? → fall back to public/_default/

Domain Map (Tier 1)

For high-traffic tenants, add a static mapping in config/static.php for O(1) lookups with zero database queries:

'domain_map' => [
    'dashboard.acme.com' => 'acme',
    'app.globex.io'      => 'globex',
],

Database Lookup (Tier 3)

Tenants whose domains aren’t in the static map and can’t be resolved by the TenantResolver are looked up via the tenants.domain column. This is handled by ChatRepository::findTenantByDomain(), which is wired in as a domainLookup callable in Application::registerStaticFileHandler().

// ChatRepository
public function findTenantByDomain(string $domain): ?array
{
    $tenant = Tenant::where('domain', $domain)->first();
    return $tenant?->toArray();
}

If you use a reverse proxy (Nginx, Caddy, ALB), configure it to pass the Host header through to Fabriq.

SPA Fallback

Single-page applications use client-side routing (React Router, Vue Router, etc.), where paths like /dashboard/settings don’t correspond to actual files on disk. When spa_fallback is enabled, the middleware serves index.html for any request that doesn’t match a static file.

GET /dashboard/settings
  → public/acme/dashboard/settings  → not a file
  → SPA fallback → public/acme/index.html  → serve (200)
  → React Router takes over client-side
API Prefixes

Requests to /api/*, /health, and /metrics are never intercepted by the SPA fallback. These are configured in api_prefixes and always pass through to the API route handlers.

Caching Strategy

The middleware applies intelligent Cache-Control headers based on file type and naming:

File TypeCache-ControlReason
HTML (.html, .htm)no-cacheAlways fetch fresh to pick up new deployments
Fingerprinted assets
app.a1b2c3d4.js		public, max-age=31536000, immutableContent hash in filename = safe to cache forever
Other static assetspublic, max-age=86400Configurable via cache_max_age

Fingerprint detection uses the regex pattern /\.[a-f0-9]{6,32}\./i, which matches filenames like app.a1b2c3d4.js, vendor.fa3e9b1c2d.css, etc.

Build Automation (CI/CD)

Fabriq includes a built-in FrontendBuilder that can clone a tenant’s Git repository, run the build command, and atomically deploy the output to public/{tenant-slug}/.

How It Works

1. Clone or pull from tenant's Git repo
     → storage/builds/{tenant-slug}/repo/

2. Checkout the configured branch

3. Run build command (default: npm install && npm run build)
     → Uses Swoole\Coroutine\System::exec() for non-blocking execution

4. Atomic deployment
     → Rename current public/{tenant-slug}/ to public/{tenant-slug}.old/
     → Copy build output (dist/) to public/{tenant-slug}/
     → Remove .old directory

5. Record BuildResult (status, commit hash, duration, log)

Configuration

// config/static.php
'build' => [
    'workspace'       => 'storage/builds',       // Git clone workspace
    'default_command' => 'npm install && npm run build',
    'default_output'  => 'dist',                  // Build output directory
    'webhook_secret'  => '',                      // Webhook signature secret
    'timeout'         => 300,                     // Build timeout (seconds)
],

Tenant Config

Each tenant stores their build configuration in their config_json:

{
  "frontend": {
    "repo": "https://github.com/acme/dashboard.git",
    "branch": "main",
    "build_command": "npm install && npm run build",
    "build_output": "dist"
  }
}

If build_command or build_output are omitted, the defaults from config/static.php are used.

CLI Commands

CommandDescription
php bin/fabriq frontend:build <slug>Clone/pull the tenant's repo, run the build, and deploy atomically. Runs synchronously in the current process.
php bin/fabriq frontend:status <slug>Show the latest build result for a tenant (status, commit, duration, timestamp).
 
# Build and deploy the "acme" tenant's frontend
php bin/fabriq frontend:build acme

# Check build status
php bin/fabriq frontend:status acme
# → status: success | commit: a1b2c3d | duration: 12340ms | 2026-02-22T10:30:00Z

Webhooks

Trigger builds automatically when code is pushed to the tenant’s repository:

MethodPathDescription
POST/api/webhooks/frontend/deployGitHub/GitLab webhook receiver

Configure your Git provider to send push events to this URL. The controller validates the webhook signature using the build.webhook_secret from config/static.php.

GitHub Webhook Setup

  1. Go to your repo → Settings → Webhooks → Add webhook
  2. Payload URL: https://api.yourplatform.com/api/webhooks/frontend/deploy
  3. Content type: application/json
  4. Secret: the same value as build.webhook_secret
  5. Events: select “Just the push event”
Security

Always set a webhook_secret in production. Without it, anyone can trigger builds by sending a POST request to your webhook endpoint.

Deployment Notes

  • Docker: if using build automation, install git, nodejs, and npm in your Dockerfile
  • Volumes: mount public/ and storage/builds/ as persistent volumes so builds survive container restarts
  • Permissions: the Swoole process needs read access to public/ and read/write access to storage/builds/
  • Reverse proxy: ensure the Host header is passed through for custom domain resolution
  • TLS: for custom domains, use a proxy that supports automatic certificate provisioning (Caddy, AWS ALB + ACM, or cert-manager on K8s)
Production without build automation

You don’t have to use Fabriq’s built-in CI/CD. You can build frontends externally (GitHub Actions, Vercel, etc.) and simply copy the output into public/{tenant-slug}/. The StaticFileMiddleware will serve them regardless of how they got there.
