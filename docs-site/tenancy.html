Multi-Tenancy

Fabriq enforces tenant isolation at the kernel level. Every HTTP request, WebSocket message, queue job, and event carries a tenant_id.

Tenant Resolution

The TenantResolver tries strategies in order until one matches. If none resolves, the request is rejected with 403 Forbidden.

StrategyHow It Works
hostExtracts subdomain from Host header (e.g., acme.myapp.com → slug acme), or matches custom domain
headerReads X-Tenant: acme header
tokenReads tenant_id claim from a decoded JWT (requires AuthMiddleware to run first)

The resolution priority is configured in config/tenancy.php:

'tenancy' => [
    'resolver_chain' => ['host', 'header', 'token'],
    'cache_ttl'      => 300,
],
1. Host subdomain  →  acme.fabriq.dev → "acme"
2. X-Tenant header →  X-Tenant: acme        → "acme"
3. JWT claim       →  { tenant_id: "acme" } → "acme"

TenantContext

Once resolved, the tenant is available everywhere via the coroutine-local Context:

use Fabriq\Kernel\Context;

$tenantId = Context::tenantId();   // UUID string

// Full tenant object (stored in extras by TenancyMiddleware)
$tenantData = Context::getExtra('tenant');
// ['id' => '...', 'slug' => 'acme', 'name' => 'Acme Corp', 'plan' => 'pro', ...]

Creating a TenantContext

use Fabriq\Tenancy\TenantContext;

// From a database row
$tenant = TenantContext::fromArray([
    'id'     => 'uuid-here',
    'slug'   => 'acme',
    'name'   => 'Acme Corp',
    'plan'   => 'pro',
    'status' => 'active',
    'domain' => 'acme.example.com',
    'config_json' => '{"feature_flags":{"dark_mode":true}}',
]);

$tenant->id;          // 'uuid-here'
$tenant->isActive();  // true
$tenant->configValue('feature_flags.dark_mode'); // true

TenantContext Properties

TenantContext {
    id: string        // UUID or generated ID
    slug: string      // URL-safe identifier
    name: string      // Display name
    plan: string      // "free", "pro", "enterprise"
    status: string    // "active", "suspended"
    config: array     // Tenant-specific overrides (including database strategy)
    dbKey: ?string    // Database routing key — drives TenantDbRouter
}
Important

Suspended tenants are rejected at resolution time. You do not need to check status in your application code.

Per-Tenant Database Routing

The TenantDbRouter dynamically selects the correct database connection based on the tenant’s configuration. This enables three isolation strategies:

StrategyHow It Works
shared (default)All tenants share the app pool. Row-level isolation via WHERE tenant_id = ?
same_serverSame MySQL server, separate database. USE <tenant_db> before each query
dedicatedTenant has its own MySQL server; a dedicated connection pool is created dynamically with LRU eviction

Configure the strategy in the tenant’s config_json:

// Shared (default — no config needed)
{ "database": { "strategy": "shared" } }

// Same server, separate database
{
  "database": {
    "strategy": "same_server",
    "name": "tenant_acme_db"
  }
}

// Dedicated server
{
  "database": {
    "strategy": "dedicated",
    "host": "10.0.1.50",
    "port": 3306,
    "name": "acme_production",
    "username": "acme_user",
    "password": "secret"
  }
}

Routing Flow

Request arrives
  → TenancyMiddleware resolves tenant
    → Context::setTenant(TenantContext)
      → Application code calls DB::table('orders')->get()
        → QueryBuilder calls TenantDbRouter::acquire('app')
          → Router inspects tenant config['database']['strategy']
            → shared:    borrow from default 'app' pool
            → same_server: borrow from 'app' pool, then USE 
            → dedicated: borrow from tenant-specific pool (LRU cached)
          → Query executes on the correct connection
        → TenantDbRouter::release() returns connection

Stored Procedures & Tenant Routing

Stored procedure calls (DB::call('sp_name')) are also routed through TenantDbRouter. If a tenant uses same_server or dedicated strategy, the CALL statement executes on the tenant’s database automatically:

// Automatically routed to the tenant's database
$result = DB::call('sp_calculate_invoice')
    ->in('order_id', $orderId)
    ->out('invoice_total')
    ->exec();

Tenant Enforcement

Tenant isolation is enforced at every layer of the system:

LayerHow tenant_id is enforced
HTTPTenancyMiddleware resolves and sets Context
WebSocketWsAuthHandler resolves from JWT, registers in Gateway
Queue JobsContext restored from job fields (tenant_id stored at dispatch)
Event ConsumerContext restored from event fields
ORM ModelsHasTenantScope trait auto-adds WHERE tenant_id = ? and injects tenant_id on insert
ORM QueryBuildertenantScoped() method adds tenant filter; models opt-in via $tenantScoped = true
DB RoutingTenantDbRouter selects pool/database per tenant strategy
RepositoryBase TenantAwareRepository adds WHERE tenant_id = ?

Tenant-Scoped Database Queries

Preferred: ORM Models. Set $tenantScoped = true on your model and all queries will automatically include WHERE tenant_id = ?:

use Fabriq\Orm\Model;

class Order extends Model
{
    protected string $table = 'orders';
    protected bool $tenantScoped = true;  // ← auto tenant isolation
}

// All queries are automatically tenant-scoped:
$orders = Order::where('status', 'pending')->get();
$order  = Order::create(['status' => 'new', 'total' => 49.99]);

Alternative: TenantAwareRepository. For raw SQL, extend TenantAwareRepository:

use Fabriq\Storage\TenantAwareRepository;

final class OrderRepository extends TenantAwareRepository
{
    public function findAll(): array
    {
        return $this->withAppDb(function ($conn) {
            $tenantId = $this->tenantId(); // throws if missing
            $stmt = $conn->prepare('SELECT * FROM orders WHERE tenant_id = ?');
            return $stmt->execute([$tenantId]);
        });
    }
}
Safety

If tenant_id is not set in the current Context, calling $this->tenantId() will throw a RuntimeException. This fail-fast behavior prevents accidental cross-tenant data access.

Config Cache

Tenant configuration is cached in-memory with a configurable TTL to avoid hitting the database on every request:

Key: sf:tenant:config:{slug}
TTL: 300 seconds (5 minutes)
$tenantCache = new TenantConfigCache(
    ttl: (float) $config->get('tenancy.cache_ttl', 300),
);

// Store a tenant in the cache
$tenantCache->put($tenant);

// Retrieve from cache (returns null if expired)
$cached = $tenantCache->get("host:acme");
